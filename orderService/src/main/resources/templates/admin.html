🚚 QR FoodTruck
팝업 푸드트럭 전용 QR 주문 시스템. 회원가입 없이 QR 스캔만으로 주문·결제·조리 상태를 실시간 확인.

주문 플로우
1
QR 접속 → 익명 세션 발급
Redis에 UUID 세션 저장. 장바구니에 첫 상품을 담는 순간 SSE 연결 시작.
2
메뉴 조회
Server A가 Redis에서 재고 직접 조회 (Read-Only). 재고 0이면 품절 표시.
3
주문 요청 → 재고 확인
재고 없으면 주문 실패 alert 반환.
4
결제창 → Toss PG
내부 PK 대신 orderNo(UUID)를 PG사에 제출. 카드 일반결제 기준.
5
결제 성공 → Redis 최종 재고 확인 → PG 최종 승인
TX 전문 저장 후 Server B에 RabbitMQ로 제조 지시.
6
Server B 제조 (7초 자동 완성)
조리 완료 시 SSE로 고객에게 Push. 관리자 화면에 주문번호 표기.
7
전달 완료 → 세션 종료
관리자가 전달완료 버튼 클릭 → 10초 후 고객 세션 만료 · 화면 초기화.
Server A — orderdb
테이블주요 컬럼비고
ordersid · customer_id · total_price · paid_price · statusstatus: ACCEPTED/PAID/PARTIAL_REFUND/FULL_REFUND
order_itemsorder_id · product_id · product_name · unit_price · quantity · status가격·상품명 스냅샷 보존
paymentsid · order_no(UUID) · order_id · total_price · paid_price · payment_type · statusorder_no → PG사 식별자. 내부 PK 외부 비노출
payment_detailspayment_id · tx_code · amount환불은 음수(-) 금액으로 기록
Server B — inventorydb
테이블주요 컬럼비고
productsid · name · price · stock_quantity · is_displayed재고·상품 통합. is_displayed로 품절 전시 제어
manufacturesid · order_id · statusACCEPTED→COMPLETED→DELIVERED / DISCARDED
모든 테이블: is_deleted · created_at · updated_at 포함 (Soft Delete). 서비스 간 FK 제약 없음 — 타입 일치만 보장.
AWS 인프라 구성
CloudFormation 4개 스택으로 인프라를 레이어별로 분리 관리. 앱 서버(ECS Fargate)와 인프라 서버(EC2)를 분리해 관심사를 명확히 구분.
인터넷 └── ALB (Application Load Balancer) ← 80포트, 경로 기반 라우팅 ├── ECS Fargate — order-service :8080 ← 주문·결제 └── ECS Fargate — inventory-service :8081 ← 재고·제조 EC2 t3.micro (인프라 전용 — docker-compose) ├── order-db MySQL :3316 ├── inventory-db MySQL :3317 ├── Redis :6379 ← 재고 캐시 · 세션 UUID └── RabbitMQ :5672 ← 서비스간 메시지 브로커

CI/CD 파이프라인 전략
GitHub main 브랜치 push 시 GitHub Actions가 자동으로 빌드 → ECR push → ECS 롤링 배포까지 처리. CloudFormation은 인프라(변경 드묾), GitHub Actions는 애플리케이션(배포마다)을 담당해 역할을 분리했다.

EC2 · ECR · ECS Cluster · ALB CloudFormation 드묾 인프라 변경은 리뷰 후 의도적으로
Task Definition · ECS Service GitHub Actions 배포마다 이미지 태그가 매번 바뀌므로 CI/CD가 적합
